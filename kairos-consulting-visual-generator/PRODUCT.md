# Product — kairos-consulting-visual-generator

这份文档讲**为什么**。事实在 `SKILL.md`，设计规范在 `references/design_system.md`。

---

## 1. 这个 skill 解决什么问题

**真正的问题不是"画图"，是"让非设计师也能产出咨询公司级别的商业视觉卡片"。**

普通用户的现实链路：

1. 写完一篇文章或做完一份报告。
2. 想做一张封面或信息图发到社交媒体。
3. 没有设计师，Canva 模板太千篇一律。
4. 让 AI 生成，结果大部分是"标题放正中 + 渐变背景 + emoji"。

我们要在第 3-4 步之间插一个东西，让产出**有灵魂、有隐喻、有克制**。

### 核心矛盾

- **质量**：要像麦肯锡报告封面或 *Monocle* 杂志卡片，不能像 PPT 模板。
- **效率**：要 10 分钟内拿到可发布的图，不能让用户在 prompt 上反复打磨。
- **不假**：数据、报价、版本号不能编。

三者之中**质量是分水岭**。Canva 解决了效率，AI 解决了速度，但都没解决"看一眼就知道是用心做的"这件事。

---

## 2. 产品定位（一句话）

> 把一个商业主题交给 AI agent，10 分钟内输出一张有隐喻、有克制、有杂志感的商业视觉卡片。

---

## 3. 目标用户

| 场景 | 用户类型 | 输入 | 期望输出 |
| --- | --- | --- | --- |
| X / 公众号封面 | 创业者 / DevRel | 主题 + 用途 | 封面卡片 |
| 小红书封面 | 内容创作者 | 主题 + 画幅 | 3:4 卡片 |
| 商业报告封面 | 咨询顾问 | 主题 + 语境 | 16:9 封面 |
| 信息图 | 产品经理 | 主题 + 数据 | 结构卡片 |
| 方法论图 | 研究员 | 主题 + 框架 | Swiss 风信息图 |

**反向画像**：

- 想做卡通 IP 风、少女风、Y2K：和两套视觉系统正面冲突，硬接出来一定难看。
- 想做复杂数据大屏：这不是数据可视化工具。
- 想自定义颜色：不允许，保护美学比给自由更重要。

---

## 4. 关键设计决策

### 决策 1：只做两套视觉系统，不做一站式

- **Editorial Magazine**：serif + 墨水 + 杂志结构
- **Swiss Consulting**：Inter + 一种 accent + 工程结构

更通用 = 更平庸。同时维护 5 种风格的结果一定是 5 种都做不好。

### 决策 2：不允许自定义颜色

从 12 套预设中选择。让用户传一个 `#A3FF21` 进来，AI 配的衍生色大概率难看。

### 决策 3：先判断信息是否足够，不盲目默认生成

Intake Gate 强制追问关键字段（用途、画幅）。信息不足时假装确定，出来的图一定是错的。

### 决策 4：单一核心商业隐喻

一张卡片只允许一个主隐喻。多个隐喻同时堆叠会让画面失控。

### 决策 5：标题必须信息重构

长主题必须拆成三层：A 层核心词、B 层完整标题、C 层极简标签。主视觉文字不能只是普通标题，必须成为画面主体结构。

### 决策 6：HTML/CSS 渲染优先于 AI 生图

文字密集的卡片优先用 HTML/CSS 渲染，保证字体、换行和遮挡可控。只有需要插画、照片质感或背景资产时才调用图像生成。

---

## 5. 视觉系统锁定

两套视觉系统的本质不可改变：

| 系统 | 视觉本质 | 禁止改变 |
| --- | --- | --- |
| Editorial Magazine | 衬线 + 墨水 + 杂志结构 + 大留白 | 不加网格、不加数据大字报、不加霓虹色 |
| Swiss Consulting | Inter + 一种 accent + 网格 + 发丝线 | 不加衬线装饰、不加水墨效果、不加渐变 |

主题预设可以新增（需验证），但视觉系统不能变成第三种风格。

---

## 6. 质感规则

输出必须像高级印刷品，不是 PPT 模板。

**必须**：平涂色块、线条清晰、留白充足、颜色克制、文字清晰

**禁止**：渐变、霓虹色、玻璃拟态、复杂阴影、油腻光效、卡通质感、商务小人

---

## 7. 产品边界

| 不做 | 原因 |
| --- | --- |
| 不允许自定义颜色 | 保护美学一致性 |
| 不做卡通/少女/Y2K 风格 | 和两套视觉系统冲突 |
| 不做复杂数据大屏 | 这是卡片，不是 dashboard |
| 不在 skill 内调用生图模型 | 生图是宿主 agent 的能力 |
| 不允许多个隐喻堆叠 | 单一隐喻是产品承诺 |

---

## 6. 反模式

1. **普通大标题**：没有文字重构的标题 → 拒绝。
2. **没有视觉隐喻**：标题旁边放图标冒充图文融合 → 拒绝。
3. **商务小人 / 握手图标 / 机器人头像** → 拒绝。
4. **科幻蓝紫背景 / 霓虹渐变** → 拒绝。
5. **5 个以上模块堆叠**：信息图失控 → 拒绝。
6. **假数据 / 假版本号 / 假报价** → 拒绝。
7. **文字被遮盖或裁切** → 拒绝。
8. **自媒体廉价感** → 拒绝。

## 7. 与 guizang-social-card-skill 的对比

| 维度 | guizang-social-card | kairos-consulting-visual |
| --- | --- | --- |
| Seed 模板 | ✅ 有 template-editorial-card.html / template-swiss-card.html | ❌ 无，Agent 从零生成 |
| 版式锁定 | ✅ 28 个 recipe（M01-M16 + S01-S12） | ✅ 11 个 layout skeleton（E01-E03 + S01-S08） |
| 颜色预设 | 10 套 | 12 套 |
| 校验脚本 | ✅ validate-social-deck.mjs（Playwright） | ✅ verify_design_system.py（静态检查） |
| 渲染管线 | Playwright 截图 → PNG | 浏览器预览 → 截图 |

**核心差距**：没有 seed 模板。Agent 从零写 HTML，一致性不可控。
