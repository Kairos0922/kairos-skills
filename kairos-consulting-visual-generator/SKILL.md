# kairos-consulting-visual-generator

## Purpose

把用户给出的主题转化为一张文字主导、隐喻清晰、结构克制的商业视觉作品。目标不是普通 PPT 模板，也不是自媒体装饰封面，而是：

- 麦肯锡式理性气质
- 高端咨询报告视觉
- 标题信息重构
- 单一核心商业隐喻
- 极简商业图形
- 清晰商业叙事

当信息充足且用户请求“直接生成画面”时，不输出分析过程和解释文字；直接调用宿主 agent 可用的图像生成能力。该 skill 不配置模型、provider、API key 或独立生图服务。

## Input Contract

先从用户输入中提取这些字段：

- `主题词 / 主标题`：必需。不得随意改变核心词、品牌名、人名、术语或英文大小写。
- `副标题`：可空。
- `画幅比例`：`5:2`、`16:9`、`4:5`、`3:4` 或 `1:1`。
- `语言`：中文、英文或中英混排。
- `用途`：决定封面逻辑或信息图逻辑。
- `补充语境`：行业背景、目标受众、使用场景、情绪倾向。
- `禁用元素`：必须遵守。

## Intake Gate

不要在需求不清楚时假装确定。先判断信息是否足以进入画面生产：

- 必须有：主题词 / 主标题、用途。
- 通常必须有：画幅比例。只有用途和平台已经明确指向唯一常见比例时，才可说明推断并继续；否则追问。
- 可推断：语言。如果标题明显是中文、英文或中英混排，可跟随输入；如果用户要求跨平台或品牌一致性但语言不清楚，追问。
- 建议补充：行业背景、目标受众、使用场景、情绪倾向。若主题泛化到会影响隐喻选择或信息层级，必须追问。
- 必须遵守：用户给出的禁用元素。若没有禁用元素，不主动追问，直接使用默认禁用清单。

信息不足时，只问 1-3 个最关键问题，不输出设计方案，不生成画面。优先追问会改变结果的字段：

1. 用途是什么：封面、信息图、咨询分析页、流程图还是矩阵图？
2. 画幅比例是什么：`5:2`、`16:9`、`4:5`、`3:4` 还是 `1:1`？
3. 这张图面向谁、用于什么场景，想表达理性、增长、风险、机会还是转型？

用户只给主题且明确说“你来定”时，可以继续，但必须先形成内部 `Visual Brief`，不得随机默认。

## Visual Brief Contract

生成前先形成一份内部视觉简报；除非用户要求，否则不要把简报完整输出。简报必须包含：

- `title_full`：用户完整主题。
- `subtitle`：用户副标题或空。
- `usage`：封面类或信息图类具体用途。
- `ratio`：最终画幅比例。
- `language`：最终语言策略。
- `audience_context`：行业、受众、场景、情绪。
- `core_word`：主视觉核心词，中文 2-8 字或英文 1-4 词。
- `title_system`：完整标题、主视觉词、2-4 个英文标签、短结论。
- `main_metaphor`：唯一主隐喻。
- `composition_model`：封面或信息图的结构模型。
- `anchor_words`：最多 2-3 个边缘锚点或模块锚点。
- `avoid`：用户禁用元素 + 默认严格禁止项。
- `qa_checks`：生成后要检查的关键风险。

更多方法论、隐喻设计语法和 QA 清单见 `references/consulting_visual_methodology.md`。

## Methodology First

稳定产出效果主要依赖视觉总监方法论，而不是脚本。不同场景的使用方式不同：

- 快速封面：只要信息足够，直接形成内部 `Visual Brief`，用封面构图和唯一主隐喻生成。
- 高价值商业报告 / PPT 封面：先做内部视觉简报和 QA 风险预判，再生成；必要时先给用户确认方向。
- 信息图 / 咨询分析页 / 方法论图：先明确阅读路径、模块数量和结论，再生成，避免把封面逻辑套到内容页。
- 流程图 / 矩阵图：用户已指定图形类型时，优先尊重指定类型；不要为了隐喻选择而改变用户要的分析结构。
- 信息不足或主题过泛：先追问，不要默认产出。

`scripts/select_metaphor.py` 只是可选辅助，用于快速检查 intake 或给出初始隐喻建议；不要把脚本结果当成不可挑战的设计判断。若脚本建议和语境冲突，以视觉简报和用户意图为准。

## Output Mode

先按用途分流：

- 封面类：X 封面、PPT 封面、商业报告封面、作品集封面、公众号封面、小红书/小绿书封面、海报封面。
- 信息图类：信息图、咨询分析页、方法论图、流程图、矩阵图。

封面只选择一个主隐喻，信息图也只围绕一个主隐喻展开多个子模块。不要多个隐喻同时堆叠。

## Metaphor Selection

可选辅助：

```bash
python3 scripts/select_metaphor.py --title "主题" --usage "用途" --context "补充语境"
```

无论是否运行脚本，都要用视觉判断选择一个最准确的主隐喻：

| 隐喻 | 适用表达 |
| --- | --- |
| 漏斗 | 筛选、转化、收敛、用户路径、销售流程、机会识别 |
| 路径 | 战略推进、方法论、增长路线、行动计划、转型进程 |
| 阶梯 | 升级、价值提升、能力进阶、商业成熟度、定价层级 |
| 矩阵 | 定位、比较、选择、竞争格局、优先级判断 |
| 坐标 | 市场地图、战略定位、机会区间、风险收益关系 |
| 飞轮 | 增长循环、复利机制、网络效应、持续运转系统 |
| 节点网络 | 系统协同、组织连接、技术架构、资源关系 |
| 数据流 | 信息流动、自动化、效率提升、输入输出关系 |
| 门槛 | 准入、筛选、风险控制、阶段关口、决策标准 |
| 窗口 | 市场机会、时间窗口、突破口、趋势入口 |
| 防线 | 风险管理、治理机制、合规控制、安全边界 |
| 断层 | 结构变化、代际变化、产业转折、认知差距 |
| 容器 | 价值承载、商业模式、资源池、收益池 |
| 罗盘 | 方向选择、战略判断、长期定位、路径导航 |
| 建筑结构 | 组织架构、能力底座、系统搭建、治理框架 |

## Title Reconstruction

长主题必须拆成三层：

- A 层：主视觉核心词。中文 2-8 字，英文 1-4 个词，巨大、可读、可被隐喻重构。
- B 层：完整标题。放在顶部信息栏、边缘信息栏或副标题区，小字号保留完整语义。
- C 层：极简标签。只用 2-4 个英文标签，如 `STRATEGY`、`MARKET INSIGHT`、`BUSINESS MODEL`、`GROWTH SYSTEM`、`TRANSFORMATION`、`INDUSTRY VIEW`、`REPORT COVER`、`INFOGRAPHIC`、`2026`。

主视觉文字不能只是普通标题。文字必须成为画面主体结构，被切割、收敛、框架化、路径化、模块化、网格化、阶梯化、矩阵化、建筑化、数据化或空间化，并与主隐喻融合。

核心词提炼规则：

- 不要把长主题整句放大。
- 不要改变用户核心概念。
- 优先提炼可承载隐喻的商业名词或动词短语。
- 避免空泛词，如“趋势”“思考”“洞察”，除非它们就是用户品牌或核心概念。
- 如果两个核心词同等重要，选择更能承载主隐喻的那个，另一个放入顶部完整标题或锚点。

## Cover Composition

封面推荐结构：

1. 顶部信息栏：完整标题、英文分类标签、年份、编号。
2. 中央主视觉区：巨型重构文字 + 核心商业隐喻。
3. 边缘概念锚点：起点词、终点词、风险词、价值词、方向词，最多 2-3 个。
4. 底部短结论：一句短判断，12-22 个中文字符以内。

封面信息密度：

- 允许：1 个巨型主视觉标题、1 个完整小标题、2-4 个英文标签、1 个核心隐喻、1 句短结论、少量细线/节点/编号/网格。
- 禁止：5 个以上模块、3 个以上图表、大段解释、密集图标、多个结论框、完整流程拆解。
- 必须留白充足，视觉中心清楚，辅助文字很少。

## Infographic Composition

信息图推荐结构：

1. 顶部标题区：完整标题、短副标题、分类标签。
2. 中央结构图区：核心流程、矩阵、系统、路径、漏斗或价值链。
3. 左右辅助模块：关键洞察、变量、约束、机会、风险。
4. 底部结论区：1-3 条关键结论或行动建议。
5. 图例与编号：极小字号、细线、浅灰辅助阅读。

信息图信息密度：

- 允许：3-6 个主要模块、1 个核心大结构、1-2 个辅助小结构、关键数据、标签、编号、注释、简短结论区。
- 禁止：8 个以上模块、过多颜色、过多图标、无层级文字、长段模块文案、数据和图形无逻辑关系。
- 阅读路径必须清晰：左到右、上到下、问题到方案、输入到输出、现状到机会、机制到结果。

## Visual System

整体接近：

- McKinsey-style report cover
- strategy consulting visual
- executive insight page
- boardroom-ready layout
- clean corporate editorial design
- minimalist business metaphor
- Swiss grid layout
- typography-led consulting design

配色：

- 主色：白色、深蓝、黑色、冷灰、浅灰。
- 强调色：只可少量使用克制蓝、极少量橙色、绿色或红色，用于关键节点、风险点、价值点、终点、决策点、重要箭头。
- 禁止蓝紫霓虹、赛博朋克光效、花哨渐变、彩虹色、高饱和广告色、大面积暖色背景。

字体：

- 中文可使用黑体、宋黑混合、现代无衬线。
- 英文使用 Helvetica / Neue Haas Grotesk / Univers / Swiss-style sans-serif 气质。
- 封面主视觉可以大，但粗细克制，不要过度粗黑。
- 信息图模块文字简短，层级明确，不要密密麻麻。

质感：

- 可用轻微纸张纹理、轻微印刷颗粒、极淡阴影、精准细线、极简网格、细边框、小型编号系统、页眉页脚、咨询报告式坐标与标注系统。
- 必须干净、克制、稳定、专业、高级、理性。

## Generation Workflow

执行顺序：

1. 提取用户输入字段。
2. 通过 `Intake Gate` 判断是否需要追问。
3. 信息足够后，形成内部 `Visual Brief`。
4. 根据用途和语境选择唯一主隐喻；必要时参考 `scripts/select_metaphor.py`。
5. 按用途选择封面或信息图构图。
6. 将核心词与主隐喻融合为画面主体。
7. 调用宿主 agent 的图像生成能力。
8. 用 `Final QA` 检查，不合格则修正提示词或重新生成。

## Final QA

生成后必须检查：

- 是否只有一个主隐喻。
- 是否保留并准确呈现用户核心词、品牌名、人名、术语和英文大小写。
- 主视觉核心词是否可读，且不是普通大标题。
- 文字和图形是否融合，而不是标题旁边放图标。
- 封面是否留白充足，信息图是否有清晰阅读路径。
- 是否出现商务小人、握手图标、卡通人物、机器人头像、科幻蓝紫背景或廉价模板感。
- 是否有中文乱码、错字、密集小字、过多模块、过多颜色或无法辨认的结构。

如果文字准确性不可靠，优先减少画面中文字，只保留用户提供的核心词、完整标题和极少量标签。

## Strict Prohibitions

禁止出现：

- 普通大标题
- 没有文字重构
- 没有视觉隐喻
- 视觉隐喻和文字割裂
- 背景装饰和主题无关
- 普通 PPT 模板感
- 商务小人
- 握手图标
- 卡通人物
- 机器人头像
- 科幻蓝紫背景
- 复杂到失控的信息图
- 内容无层级
- 字体过粗或过花
- 主标题不可读
- 画面过满
- 自媒体廉价感

## Final Quality Bar

最终作品必须：

1. 视觉上像顶级咨询公司作品。
2. 主题表达清晰。
3. 标题经过信息重构。
4. 图形和文字融合。
5. 封面版本简洁有记忆点。
6. 信息图版本丰富但有秩序。
7. 麦肯锡式专业、理性、克制。
8. 可以直接用于 X 封面、PPT 封面、商业报告封面、咨询信息图或日常平台封面。
9. 即使去掉大部分装饰，核心结构仍然成立。
