# kairos-consulting-visual-generator

## Purpose

把用户给出的主题转化为一张文字主导、隐喻清晰、结构克制但有灵魂的商业视觉卡片。目标不是普通 PPT 模板，也不是自媒体装饰封面，更不是把信息图模块堆满，而是：

- 精致的社交卡片和杂志感卡片
- 麦肯锡式理性气质
- 高端咨询报告视觉
- 标题信息重构
- 单一核心商业隐喻
- 极简商业图形
- 清晰商业叙事和情绪记忆点

当信息充足且用户请求“直接生成画面”时，不输出分析过程和解释文字；直接调用宿主 agent 可用的图像生成能力。该 skill 不配置模型、provider、API key 或独立生图服务。

## Input Contract

先从用户输入中提取这些字段：

- `主题词 / 主标题`：必需。不得随意改变核心词、品牌名、人名、术语或英文大小写。
- `副标题`：可空。
- `画幅比例`：`5:2`、`16:9`、`4:5`、`3:4` 或 `1:1`。
- `语言`：中文、英文或中英混排。
- `用途`：决定封面逻辑或信息图逻辑。
- `补充语境`：行业背景、目标受众、使用场景、情绪倾向。
- `禁用元素`：必须遵守。

## Intake Gate

不要在需求不清楚时假装确定。先判断信息是否足以进入画面生产：

- 必须有：主题词 / 主标题、用途。
- 通常必须有：画幅比例。只有用途和平台已经明确指向唯一常见比例时，才可说明推断并继续；否则追问。
- 可推断：语言。如果标题明显是中文、英文或中英混排，可跟随输入；如果用户要求跨平台或品牌一致性但语言不清楚，追问。
- 建议补充：行业背景、目标受众、使用场景、情绪倾向。若主题泛化到会影响隐喻选择或信息层级，必须追问。
- 必须遵守：用户给出的禁用元素。若没有禁用元素，不主动追问，直接使用默认禁用清单。

信息不足时，只问 1-3 个最关键问题，不输出设计方案，不生成画面。优先追问会改变结果的字段：

1. 用途是什么：封面、信息图、咨询分析页、流程图还是矩阵图？
2. 画幅比例是什么：`5:2`、`16:9`、`4:5`、`3:4` 还是 `1:1`？
3. 这张图面向谁、用于什么场景，想表达理性、增长、风险、机会还是转型？

用户只给主题且明确说“你来定”时，可以继续，但必须先形成内部 `Visual Brief`，不得随机默认。

## Visual Brief Contract

生成前先形成一份内部视觉简报；除非用户要求，否则不要把简报完整输出。简报必须包含：

- `title_full`：用户完整主题。
- `subtitle`：用户副标题或空。
- `usage`：封面类或信息图类具体用途。
- `ratio`：最终画幅比例。
- `language`：最终语言策略。
- `audience_context`：行业、受众、场景、情绪。
- `core_word`：主视觉核心词，中文 2-8 字或英文 1-4 词。
- `title_system`：完整标题、主视觉词、2-4 个英文标签、短结论。
- `main_metaphor`：唯一主隐喻。
- `content_architecture`：信息卡必须先提炼内容结构，再进入版式；不能用泛化模块充数。
- `content_density`：封面、轻信息卡、标准信息卡、深信息卡的密度等级。
- `type_reconstruction`：文字重构动作，必须服务主隐喻。
- `visual_system`：`Editorial Magazine` 或 `Swiss Consulting`，只能选一个主导系统。
- `theme_preset`：固定主题色预设，不临场自定义颜色。
- `font_system`：中文显示字、正文字、英文标签的字重和字号层级。
- `layout_skeleton`：从固定骨架中选择，不从空白画布自由堆叠。
- `safe_area`：画幅、边距、底栏和文字安全区。
- `composition_model`：封面或信息图的结构模型。
- `anchor_words`：最多 2-3 个边缘锚点或模块锚点。
- `avoid`：用户禁用元素 + 默认严格禁止项。
- `qa_checks`：生成后要检查的关键风险。

更多方法论、设计系统、隐喻设计语法和 QA 清单见：

- `references/design_system.md`：字体、颜色、网格、版式骨架和渲染纪律。
- `references/consulting_visual_methodology.md`：工作流、隐喻语法、文字重构和 QA。

## Card First Principle

默认把产物当作“可传播的精致卡片”来设计，而不是普通信息图或 PPT 页面。即使用户说“信息图”，也要先保证它是一张好看的卡片，再承载结构信息。注意：卡片感不是低信息量；信息卡必须有足够内容价值，只是不能失控堆叠。

核心原则：

- 卡片优先：一屏只表达一个核心判断，但必须承载与主题匹配的信息量，不能用空泛三模块假装信息图。
- 灵魂优先：必须有一个让人记住的主句、张力或观点，不只是概念罗列。
- 版式优先：先选择版式骨架，再填内容；不要边写边堆模块。
- 字体优先：大字要有呼吸感，越大越不能粗；中文不得被遮盖、挤压或变形。
- 质感优先：用留白、字距、细线、纸感、图文关系建立高级感，不靠阴影、卡片套卡片和廉价渐变。
- 约束优先：颜色、字体、布局不能随意发散；不要让用户选择一堆设计参数。

## Content Architecture First

信息图、咨询分析页、方法论图、流程图和矩阵图必须先做内容架构，再做视觉布局。不要先套模板再填泛化文案。

如果用户输入是 URL、文章标题、研究报告、长文本或具体事件：

- 必须先读取或提炼原文关键信息；不能只根据标题臆造模块。
- 提取 6-10 个 `content_atoms`：核心论点、背景变化、机制、变量、冲突、证据、风险、行动建议、结果。
- 合并成 3-6 个视觉模块，每个模块必须有具体信息，不得写成“流程重设计”“标准成为杠杆”这类可套任何主题的空话。
- 模块标题要反映内容事实或判断，例如“AI changes planning cadence”，而不是“流程优化”。
- 如果没有足够原文或语境，必须追问或先做资料读取；不要默认产出。

内容密度按用途选择：

| 类型 | 适用场景 | 内容要求 |
| --- | --- | --- |
| `Cover` | X 封面、PPT 封面、报告封面 | 1 个主句 + 2-3 个锚点 + 1 个短结论 |
| `Light Card` | 观点卡、公众号头图、小红书封面 | 1 个主句 + 3-4 个信息点 |
| `Standard Infographic` | 信息图、咨询分析页、方法论图 | 1 个中心结构 + 4-6 个模块 + 1-2 条结论 |
| `Deep Analysis Card` | 报告页、矩阵图、流程图、复杂文章 | 1 个中心结构 + 5-7 个模块 + 变量/约束/行动建议 |

内容越多，主视觉字号越克制；不能把封面级大字和深信息卡密度同时塞进同一张图。

## Methodology First

稳定产出效果主要依赖视觉总监方法论，而不是脚本。不同场景的使用方式不同：

- 快速封面：只要信息足够，直接形成内部 `Visual Brief`，用封面构图和唯一主隐喻生成。
- 高价值商业报告 / PPT 封面：先做内部视觉简报和 QA 风险预判，再生成；必要时先给用户确认方向。
- 信息图 / 咨询分析页 / 方法论图：先明确内容架构、阅读路径、模块数量和结论，再把它组织成杂志感结构卡片；避免普通 PPT 信息堆叠，也避免信息量过低。
- 流程图 / 矩阵图：用户已指定图形类型时，优先尊重指定类型；不要为了隐喻选择而改变用户要的分析结构。
- 信息不足或主题过泛：先追问，不要默认产出。

`scripts/select_metaphor.py` 只是可选辅助，用于快速检查 intake 或给出初始隐喻建议；不要把脚本结果当成不可挑战的设计判断。若脚本建议和语境冲突，以视觉简报和用户意图为准。

## Output Mode

先按用途分流：

- 封面卡片类：X 封面、PPT 封面、商业报告封面、作品集封面、公众号封面、小红书/小绿书封面、海报封面。
- 结构卡片类：信息图、咨询分析页、方法论图、流程图、矩阵图。

封面只选择一个主隐喻，信息图也只围绕一个主隐喻展开多个子模块。不要多个隐喻同时堆叠。

## Visual Systems

只在两套视觉系统中选择，不要临场发明风格：

- `Editorial Magazine`：杂志感、叙事、观点、人物/组织/文化主题。特征是大留白、强主句、宋黑混排或优雅无衬线、少量细节标注、纸张质感。
- `Swiss Consulting`：分析、框架、方法论、产品、技术组织主题。特征是网格、极强字号对比、发丝线、单一锚点色、克制模块。

选择规则：

- 如果主题强调观点、文化、组织气质、趋势判断，优先 `Editorial Magazine`。
- 如果主题强调流程、方法、矩阵、系统、机制，优先 `Swiss Consulting`。
- 如果两者都适合，用 `Editorial Magazine` 做情绪和主句，用 `Swiss Consulting` 管结构；但画面只允许一个主导系统。

具体字体、色板、网格和骨架规则见 `references/design_system.md`。执行时必须先选视觉系统、主题色和版式骨架，再开始写 HTML/CSS。

## Locked Palette

不要让用户自由选色。根据 `references/design_system.md` 选择一套固定 `theme_preset`；颜色必须通过 CSS variables 管理，避免 inline 随机色：

- `Ink Classic`：暖白、墨黑、冷灰。适合高级封面、观点卡。
- `Indigo Porcelain`：瓷白、雾蓝、深蓝、银灰。适合科技、AI、产品。
- `Graphite Red`：炭黑、纸白、灰、少量红。适合风险、治理、判断。
- `Olive Editorial`：暖纸、深橄榄、黑、浅灰。适合组织文化、长期主义。
- `IKB`：国际克莱因蓝、白、黑、浅灰。适合咨询分析、方法论、技术组织。
- `Safety Orange`：橙、白、黑、灰。适合风险、预警、产业变化、新闻紧迫感。

每张卡片最多一个强调色。不要紫蓝科幻渐变，不要多彩信息图。

默认规则：

- 不知道选什么时，Editorial 用 `Ink Classic`，Swiss 用 `IKB`。
- AI、科技、研究优先 `Indigo Porcelain` 或 `IKB`。
- 风险、治理、判断优先 `Graphite Red` 或 `Safety Orange`。
- 组织文化、长期主义优先 `Forest Ink` 或 `Olive Editorial`。

## Title Reconstruction

长主题必须拆成三层：

- A 层：主视觉核心词。中文 2-8 字，英文 1-4 个词，巨大、可读、可被隐喻重构。
- B 层：完整标题。放在顶部信息栏、边缘信息栏或副标题区，小字号保留完整语义。
- C 层：极简标签。只用 2-4 个英文标签，如 `STRATEGY`、`MARKET INSIGHT`、`BUSINESS MODEL`、`GROWTH SYSTEM`、`TRANSFORMATION`、`INDUSTRY VIEW`、`REPORT COVER`、`INFOGRAPHIC`、`2026`。

主视觉文字不能只是普通标题。文字必须成为画面主体结构，被切割、收敛、框架化、路径化、模块化、网格化、阶梯化、矩阵化、建筑化、数据化或空间化，并与主隐喻融合。

可用的文字重构动作包括：

- 拉伸
- 压缩
- 切割
- 镂空
- 错位
- 分层
- 模块化
- 框架化
- 路径化
- 收束化
- 阶梯化
- 矩阵化
- 建筑化
- 数据化
- 图表化
- 空间化

核心词提炼规则：

- 不要把长主题整句放大。
- 不要改变用户核心概念。
- 优先提炼可承载隐喻的商业名词或动词短语。
- 避免空泛词，如“趋势”“思考”“洞察”，除非它们就是用户品牌或核心概念。
- 如果两个核心词同等重要，选择更能承载主隐喻的那个，另一个放入顶部完整标题或锚点。

语义重构映射：

- 增长：扩张、上升、飞轮、复利结构。
- 转化：收敛、筛选、漏斗、路径结构。
- 战略：路径、坐标、分叉、决策结构。
- 系统：模块、链路、节点、架构结构。
- 风险：阈值、防线、断层、分级结构。
- 市场：坐标、窗口、区块、竞争格局。
- 价值：阶梯、刻度、容器、交换结构。
- 效率：压缩、加速、流动、自动化结构。
- 机会：开口、窗口、突破点、增长通道。
- 组织：层级、重组、协作网络、治理结构。

图文融合动作：

- 商业图形穿过字形。
- 路径线进入文字内部。
- 漏斗结构藏在文字负空间中。
- 矩阵网格成为字的骨架。
- 阶梯成为字的底座。
- 数据流从字中穿出。
- 断层切开文字。
- 节点网络连接笔画。
- 坐标系统嵌入标题。
- 文字变成商业结构本身。

严禁把普通标题旁边放一个图标，或在普通标题下方放一个独立图表来冒充图文融合。

## Cover Composition

封面卡片推荐结构：

1. 顶部信息栏：完整标题、英文分类标签、年份、编号。
2. 中央主视觉区：巨型重构文字 + 核心商业隐喻。
3. 边缘概念锚点：起点词、终点词、风险词、价值词、方向词，最多 2-3 个。
4. 底部短结论：一句短判断，12-22 个中文字符以内。

封面信息密度：

- 允许：1 个巨型主视觉标题、1 个完整小标题、2-4 个英文标签、1 个核心隐喻、1 句短结论、少量细线/节点/编号/网格。
- 禁止：5 个以上模块、3 个以上图表、大段解释、密集图标、多个结论框、完整流程拆解。
- 必须留白充足，视觉中心清楚，辅助文字很少。

## Infographic Composition

结构卡片推荐结构：

1. 顶部标题区：主句、完整标题、分类标签。
2. 中央主体区：一个大结构或一个强视觉隐喻。
3. 内容模块区：根据密度承载 4-6 个具体内容原子，服务主句。
4. 底部收束区：1-2 条关键结论或行动建议。
5. 图例与编号：极小字号、细线、浅灰辅助阅读。

信息图信息密度：

- 允许：1 个核心大结构、4-6 个具体内容模块、少量标签、编号、注释、1-2 条结论区。
- 禁止：8 个以上模块、少于 4 个具体内容原子、过多颜色、过多图标、无层级文字、长段模块文案、数据和图形无逻辑关系、模块把画面切碎。
- 阅读路径必须清晰：左到右、上到下、问题到方案、输入到输出、现状到机会、机制到结果。

## Visual System

整体接近：

- McKinsey-style report cover
- strategy consulting visual
- executive insight page
- boardroom-ready layout
- clean corporate editorial design
- minimalist business metaphor
- Swiss grid layout
- typography-led consulting design

配色：

- 主色：白色、深蓝、黑色、冷灰、浅灰。
- 强调色：只可少量使用克制蓝、极少量橙色、绿色或红色，用于关键节点、风险点、价值点、终点、决策点、重要箭头。
- 禁止蓝紫霓虹、赛博朋克光效、花哨渐变、彩虹色、高饱和广告色、大面积暖色背景。

字体：

- 字体层级必须使用 `references/design_system.md` 中的 token，不临场随意定字号。
- 字体系统必须先选 `Swiss Sans`、`Editorial Serif` 或 `Hybrid Report` 配方；不要一套中文字体从主标题用到正文。
- 中文优先使用系统中可靠的现代黑体或宋黑混排；不要使用默认浏览器粗黑导致笨重。
- 英文使用 Helvetica / Neue Haas Grotesk / Univers / Swiss-style sans-serif 气质。
- 大字越大，字重越轻；超过 56px 的中文标题在咨询风中通常不要超过 600 weight。
- AI、工程、产品类信息卡默认用轻字重现代黑体；文化、组织、长期主义、报告封面可用宋黑混排增强杂志感。
- 标题行高必须 >= 1.08，正文行高必须 >= 1.35。
- 任意文字块必须留出安全边距，不得被底栏、图形、卡片边框或截图裁切遮盖。
- 信息图模块文字简短，层级明确，不要密密麻麻。

质感：

- 可用轻微纸张纹理、轻微印刷颗粒、极淡阴影、精准细线、极简网格、细边框、小型编号系统、页眉页脚、咨询报告式坐标与标注系统。
- 必须干净、克制、稳定、专业、高级、理性。

## Generation Workflow

执行顺序：

1. 提取用户输入字段。
2. 通过 `Intake Gate` 判断是否需要追问。
3. 信息足够后，形成内部 `Visual Brief`。
4. 对信息卡先做 `content_architecture`：提炼内容原子、分组、阅读路径、结论。
5. 根据用途和语境选择唯一主隐喻；必要时参考 `scripts/select_metaphor.py`。
6. 按用途和内容密度选择封面或信息图构图。
7. 将核心词与主隐喻融合为画面主体；信息卡必须让结构承载内容，不做空泛装饰。
8. 从 `references/design_system.md` 选主题、字体层级、网格和骨架。
9. 对文字密集卡片，优先用单文件 HTML/CSS 渲染 PNG，保证字体、换行和遮挡可控；不要用纯 SVG 手工堆坐标做复杂中文排版。
10. 只在需要插画、照片质感或背景资产时调用宿主 agent 的图像生成能力；精确中文文字仍由 HTML/CSS 承载。
11. 用 `Final QA` 检查，不合格则修正布局或重新生成。

## Final QA

生成后必须检查：

- 是否只有一个主隐喻。
- 是否保留并准确呈现用户核心词、品牌名、人名、术语和英文大小写。
- 主视觉核心词是否可读，且不是普通大标题。
- 主视觉文字是否经过明确重构动作，而不是只调整字号和字重。
- 文字和图形是否融合，而不是标题旁边放图标。
- 封面是否留白充足，信息图是否有清晰阅读路径。
- 是否出现商务小人、握手图标、卡通人物、机器人头像、科幻蓝紫背景或廉价模板感。
- 是否有中文乱码、错字、密集小字、过多模块、过多颜色或无法辨认的结构。
- 是否有文字被遮盖、贴边、截断，或行高/字重导致中文显脏。
- 信息卡是否有足够具体内容，而不是只有 3 个泛化模块。
- 模块内容是否来自用户输入、原文或明确语境，而不是套话。
- 线条、装饰图形、节点、边框是否进入文字安全带。
- 是否像一张精致卡片，而不是普通 PPT 页、后台面板或密集说明图。

如果文字准确性不可靠，优先减少画面中文字，只保留用户提供的核心词、完整标题和极少量标签。

## Strict Prohibitions

禁止出现：

- 普通大标题
- 没有文字重构
- 没有视觉隐喻
- 视觉隐喻和文字割裂
- 背景装饰和主题无关
- 普通 PPT 模板感
- 商务小人
- 握手图标
- 卡通人物
- 机器人头像
- 科幻蓝紫背景
- 复杂到失控的信息图
- 后台仪表盘式信息图
- 机械堆叠的流程框
- 粗笨中文大黑体
- 文字被遮盖或裁切
- 内容无层级
- 字体过粗或过花
- 主标题不可读
- 画面过满
- 自媒体廉价感

## Final Quality Bar

最终作品必须：

1. 视觉上像顶级咨询公司作品。
2. 同时像一张可传播的精致杂志卡片。
3. 主题表达清晰。
4. 标题经过信息重构。
5. 图形和文字融合。
6. 封面版本简洁有记忆点。
7. 结构卡片丰富但不拥挤。
8. 可以直接用于 X 封面、PPT 封面、商业报告封面、咨询信息图或日常平台封面。
9. 即使去掉大部分装饰，核心结构仍然成立。
