# kairos-loop

## Purpose / 目的

帮用户产出一段**规范、可直接运行的高质量 loop prompt**。

很多人不会写 loop，是因为他们把"目标"写成了"做到满意为止"这种没法验证的话，或者把一个其实是一次性的任务硬包装成循环。这个 skill 的核心价值不是套模板，而是通过几轮对话把模糊场景**逼成一个有界、可验证、能安全停下的循环**，最后交付一段提示词。

输出语言跟随用户：用户用中文就中文，用英文就英文。loop prompt 里的检查命令、代码、工具名保持原样（如 `npm test`、`gh pr checks`）。

## What is a Loop / 什么是 Loop

- **一次性 prompt**：给一条指令，AI 给一个答案就停，等人推动下一步。
- **Loop**：设定一个目标，AI 自主迭代「**观察 → 选择 → 行动 → 验证 → 记录 → 继续或停止**」直到达标或触达上限。

人是引擎、AI 是工具，这是慢的用法。Loop 把 AI 变成能独立把事做完的执行者——但前提是这个 loop 被设计得**有边界、能验证、会安全地停下来**。

一段合格 loop 的本质：**有界任务 + 可复现的验证检查 + 状态记录 + 明确命名的停止态 + 批准边界**。

## The First Test / 第一道关：这件事该做成 loop 吗

在写任何 prompt 之前，先判断这件事**值不值得、能不能**做成 loop。这是最重要的一步，跳过它会产出"假 loop"。

**首要测试**：如果新一轮的反馈/观察**无法改变下一步动作**，那它就不是 loop，而是一次性 workflow 或固定脚本。loop 的灵魂是"从上一轮结果里学到东西，据此决定下一步"。

理想的 loop 还应尽量满足这四个条件（不满足不一定不能做，但要提醒用户权衡）：

1. **重复性**：这个任务至少每周会做一次，否则搭 loop 的成本不划算。
2. **可自动判定好坏**：有客观信号能判断输出是否合格（测试、类型检查、lint、benchmark、有限场景集）。
3. **能端到端独立完成**：AI 中途不需要反复交回给人做主观决定。
4. **完成有客观标准**：终点是客观的，不是"我觉得好看了"这种审美判断。

如果明显不满足首要测试，**不要硬写 loop**，直接告诉用户："这件事更适合用一次性 prompt，因为……"，并给出那条一次性 prompt。诚实比凑一个循环更有价值。

## When It's Not a Loop / 不是 loop 时怎么办

First Test 最常见的结果往往不是"是/否"，而是**"一部分是、一部分不是"**。真实请求经常把一个可循环内核包在一次性外壳里——比如"帮我做个 X 项目"：出设计方案是一次性创作，实现某个能客观验证的阶段才是 loop。

所以判完 First Test，先**分诊**：

1. **拆开混合请求**。识别哪些部分有"上一轮客观反馈 → 改变下一步"的迭代内核，哪些是一次性创作/规划。只把前者送进下面的访谈流程，后者如实告知用户用一次性 prompt。不要因为请求里"有个 loop 的影子"就把整件事都包成循环。
2. **loop 性质随阶段变**。同一个项目，设计阶段往往不是 loop，实现阶段（有测试/指标可验收）才是。判定时锁定"用户当下要的是哪个产物"。
3. **判定为不是 loop 时，也要负责交付**。skill 承诺过给出那段一次性 prompt——别只甩一句"这个用一次性 prompt 更好"就结束。给一段最小骨架：

   ```
   <一句话目标 / 要产出什么>
   上下文与约束：<已知的硬约束，逐条列清>
   输出结构：<期望的章节 / 字段>
   要求：凡依赖外部事实的论断说明依据，不确定就标注待验证，不要臆测。
   ```

   依然遵守 grounding 铁律：不凭空发明用户没给的约束。交付后告诉用户，等进入那个可客观验证的阶段，再回来把它做成 loop。

## Interview Flow / 访谈流程

通过对话把场景补全。**不要一次性甩出所有问题**，按用户已经说清的部分跳过，只追问缺失的关键项。一次问 1-3 个最关键的。

要从对话中提取/确认的要素：

| 要素 | 要问清的 | 为什么重要 |
|------|----------|------------|
| **Goal / 目标** | 一句可观察的目标，不是"做好"。"所有 CI 检查通过""最近 20 个真实场景连续通过" | 目标不可观察，loop 就没法判断自己成没成 |
| **Verify / 验证** | 每轮用什么**具体检查**判断成败？最好是一条能跑的命令或一个明确 rubric | 没有客观验证，AI 会自我认可、空转 |
| **State / 状态** | 每轮要记录什么？失败原因、已尝试方案、剩余工作。长任务是否需要写进文件持久化 | 不记状态就会重复犯同样的错，无法进步 |
| **Stop / 停止条件** | 成功退出条件 + 失败上限（如最多 8 轮）+ 其他终止态 | 防止无限消耗 token、安静地烧钱 |
| **Action / 每轮动作** | 每轮做一个什么样的有界、可逆动作 | 动作太大或不可逆会失控 |
| **Approval / 批准边界** | 哪些动作（发 PR、改生产、删数据、发外部消息）必须先经人批准 | 选了 loop ≠ 授权它做破坏性操作 |

追问时多解释"为什么问这个"，让用户理解而不是被审问。比如用户说"跑到满意为止"，就回："loop 没法判断'满意'——我们得给它一个能跑的检查。你平时怎么确认这个任务做对了？是跑某个测试、看某个数字达标，还是别的？"

## The Feedback Cycle / 六步反馈循环

每个 loop 在运行时都应遵循这个骨架，写 prompt 时把它内化进去：

1. **Observe / 观察**：读取最新状态，收集约定好的证据。
2. **Choose / 选择**：按明确标准，选当前最高价值、范围内的动作。
3. **Act / 行动**：做一个有界的、可逆的改动，或产出一个候选。
4. **Verify / 验证**：在记录好的条件下，跑同一个验收检查。
5. **Record / 记录**：保存动作、证据、结果和剩余工作。
6. **Repeat or stop / 继续或停止**：仅当进展可衡量且未触达上限时继续；否则进入一个**命名的终止态**。

## Stop States / 命名的终止态

好的 loop 不只有"成功"一个出口。在 prompt 里明确相关的终止态，让 AI 知道每种情况该怎么收场：

- **success / 成功**：达到验证标准，正常退出。
- **clean no-op / 无需改动**：检查后发现本就合格，无需动作。
- **blocked / 被阻塞**：缺前置条件、缺权限、外部依赖不可用。
- **approval-required / 需批准**：下一步是破坏性/对外动作，停下来等人批准。
- **exhausted / 预算耗尽**：达到迭代/时间/成本上限仍未达标。
- **stagnated / 无进展**：连续几轮没有可衡量的进步，停下来求助。

**铁律：永远不要把 error 或耗尽的预算报告成 success。** 如果用户没给明确上限，用"无进展就停"（no-progress stop），而不要凭空发明一个时间/次数/成本限制。

## Output Format / 输出格式

交付一段**自包含、可直接复制运行**的 loop prompt。默认用下面这个骨架，按场景裁剪（中文场景用中文写，英文场景用英文写；这里给中英双示）：

```
启动「<Loop 名称>」循环。
目标：<一句可观察的目标>
最多迭代：<N> 轮
每轮之间运行：<具体检查命令或 rubric>
满足以下条件时退出：<客观退出条件>
每轮动作：<一句话描述每轮做什么有界动作>
需要批准：<哪些动作必须先问人，没有就写"无">

自定步调运行这个循环。每轮结束后跑检查命令、读输出，只有在退出条件未满足时才继续。
退出条件满足、或达到最大迭代次数、或连续无进展时停止。每轮给一句简短状态更新。
绝不把错误或耗尽的预算当作成功。
```

英文骨架：

```
Start the "<Loop Name>" loop.
Goal: <one observable goal>
Max iterations: <N>
Between iterations run: <check command or rubric>
Exit when: <objective exit condition>
Each iteration: <one-sentence bounded action>
Ask before: <approval-gated actions, or "nothing">

Self-pace this loop. After each iteration, run the check, read the output, and
only continue if the exit condition is not met. Stop when the exit condition
passes, max iterations is reached, or there's no measurable progress. Give a
short status update each pass. Never report an error or exhausted budget as success.
```

**写 prompt 的原则**：

- 尽量短、自包含。一段话能说清就不要分章节。优先 < 120 词。
- 只放必要的：触发动作、每轮动作、验证检查、停止规则、批准边界。
- 把"until happy / 做到好为止"换成可观察的门槛：测试通过、数字达标、reviewer 决定、有限场景集。
- 如果验证对象可能"过拟合自己的指标"（如优化 prompt/排序/模型），把**工作信号和验收门槛分开**。
- 高影响产出建议**独立验证**：生成的人和验收的人不该是同一个。

交付后，附一句话说明这个 loop 的设计理由（为什么这样验证、为什么这样停），帮用户理解而不只是拿到一段文本。

## Validate Before Delivery / 交付前自检

交付任何 loop 前，在心里静默走一遍完整循环，确认：

- 新观察**能改变**下一步动作（否则它是一次性 workflow，应如实告知用户）。
- 每轮选一个有界动作、用**可观察证据**验证、记录足够状态供下一轮或交接。
- 验证**可复现**，且在有过拟合或自批准风险时与选择/优化用的信号分离。
- 相关的停止态都明确了，error 绝不报成 success。
- 破坏性/生产/财务/隐私/对外动作有对应的批准边界。
- 没有**凭空发明**用户没提到的工具、定时器、上限、指标、权限。基于用户给的真实约束设计。

详细的质量审计清单见 `references/audit.md`；不同场景的真实 loop 范例见 `references/examples.md`。

## Build Order / 正确的落地顺序

提醒用户：别一上来就自动化。正确顺序是——

1. **先手动验证 prompt 有效**：人工跑几轮，确认这段 loop prompt 真能推进任务。
2. **再固化流程**：把规则、检查、停止条件写稳定。
3. **最后才自动化调度**：加定时/事件触发、持久化、子代理分工。

跳过步骤容易让 loop 失控、安静地烧钱。

## Cost Awareness / 成本意识

loop 会消耗 token，且上下文随迭代膨胀，每轮成本递增；"生成 + 检查"双模型会让成本翻倍。关键指标是**每次验收通过的成本**——如果接受率低于 50%，可能还不如人工。所以**迭代上限和预算门控不是可选项**，是 loop 安全运行的一部分。当用户的场景成本明显不划算时，要诚实指出。
