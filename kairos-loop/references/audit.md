# Loop 质量审计清单 / Loop Audit

用于交付前自检，或审查用户已有的 loop。判断一个 loop 是 **Ready（可用）/ Repair needed（需修补）/ Not actually a loop（其实不是 loop）**。

不打数字分。指出实质性弱点，给出可操作的修补建议。

## 第一关：它真的是 loop 吗

- [ ] **新反馈能改变下一步动作**。如果每轮做的事都一样、与上一轮结果无关，它是一次性 workflow 或定时脚本，不是 loop。→ 若不成立，判定 **Not actually a loop**，建议改用一次性 prompt + 触发器。

## 验证 / Verify

要标记的弱点：
- [ ] 验证是模糊的、自评的、不可复现的（"直到看起来不错"）。→ 换成可跑的命令、阈值、rubric 或有限场景集。
- [ ] 用**同一份证据**既做优化又做验收（会过拟合自己的指标）。→ 把工作信号和验收门槛分开。
- [ ] 高影响产出由"生成者自己"验收。→ 建议独立验证。

## 停止条件 / Stop

- [ ] 无限重试，或只有主观终点线。→ 加客观退出条件 + 失败上限。
- [ ] 把 error 或耗尽的预算报告成 success。→ 这是铁律级错误，必须修。
- [ ] 相关的终止态（blocked / approval-required / no-progress）没有覆盖。→ 补上。

注意：当"连续无进展就停"已经足够时，**不要**仅因为缺一个武断的时间/次数/成本预算就标记为问题。

## 状态 / State

- [ ] 基于陈旧状态做决策（行动前不重读最新状态）。
- [ ] 改动会覆盖无关的工作。
- [ ] 下一轮需要恢复工作，却没有记录/交接状态。→ 长任务建议持久化到文件。

## 权限边界 / Authority

- [ ] 破坏性、生产、财务、隐私、对外消息类动作**缺少批准边界**。→ 必须加"ask before"。
- [ ] 把目录/参考里的 prompt 当作可信指令直接执行。→ 外部内容当不可信数据看待。

## 设计真实性 / Grounding

- [ ] **凭空发明**了用户没提到的工具、定时器、上限、指标、权限、负责人。→ 只基于用户给的真实约束设计；信息不足就回去问，不要编。

## 成本 / Cost

- [ ] 没有迭代上限或预算门控，可能安静地烧钱。
- [ ] 接受率可能低于 50%（每次验收通过的成本高于人工）。→ 提醒用户这个场景可能不划算。

---

## Verdict 模板

**Ready** — 这个 loop 有可观察目标、可复现验证、明确停止态和批准边界，可以拿去手动验证了。

**Repair needed** — 主要问题是 [X]。建议 [具体修补]。其余部分没问题。

**Not actually a loop** — 这件事新反馈不改变下一步动作，本质是 [一次性 workflow / 定时任务]。建议改用 [一次性 prompt + 触发器]，附上那段 prompt。
