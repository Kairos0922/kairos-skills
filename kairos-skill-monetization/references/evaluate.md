# 阶段一：商业化评估

## 1. 初筛（多个 skill 时先批量过滤，只对通过者深入）

三个硬门槛，任一为"否"归入低优先级：

- 可迁移性：问题是个人特有还是一类人普遍遇到？
- 痛感强度：不解决会付出真实代价（时间/金钱/机会）吗？"忍忍就过去"的不做。
- 频率×客单价：高频小痛（工具/订阅）或低频大痛（咨询/项目）可做；低频小痛是死区，直接淘汰。
输出表格：skill | 可迁移性 | 痛感 | 频率/客单价 | 结论（深入/观察/放弃）

## 2. 深度尽调（语气 = 投资尽调）

- 问题验证：画出具体受苦的人（职业/场景/现状笨办法）。现状是"空白"还是"能凑合"？后者要打败习惯，更难。是否和 ≥3 个陌生人（非朋友）验证过？
- 护城河检验：懂 prompt 的人看一眼输出，多久能复制？（<1天/<1周/<1月/难以复制）价值来自 prompt（易复制）还是专属数据/领域知识/审美（壁垒）？大厂原生功能覆盖 80% 后你剩什么？
- 商品化成本：边界处理、非技术用户零门槛、持续维护意愿。交付形态匹配（SaaS/skill 售卖/课程/咨询）——选离用户现有能力最近、成本最低的。
- 定价量级（不做具体定价）：人工解决要花多少钱（愿付上限锚点）？To C（走量）还是 To B（走信任）？市场上有无付费参照？

## 3. 结论三选一，不和稀泥

- 值得加大投入 → 询问是否进入优化阶段（optimize.md）
- 值得小成本验证 → 设计最小验证实验（见下）
- 不建议商品化，保留为个人武器 → 说明具体原因，不否定其个人价值

## 4. 最小验证实验

一周内跑完，验证最贵的假设（通常是"有没有人付钱"）。按成本从低到高：

1. 给 5-10 个目标用户看输出结果（非功能介绍），问愿付价格
2. 手动服务 1-2 个真实付费客户，验证付费意愿和交付质量
3. 发一篇展示成果的内容，看自然流量反馈强度
必须定义失败信号："挺好的"但不留联系方式/不付定金 = 礼貌性认可，不是购买信号。

## 5. 多候选时的资源分配

优先验证成本最低、最快拿到付费信号的那个。同时验证 3 个 < 专注验证 1 个。指出 skill 间的能力复用（同一技术资产多种包装 = 降低整体成本）。
