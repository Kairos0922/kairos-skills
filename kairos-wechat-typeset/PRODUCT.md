# Product — kairos-wechat-typeset

这份文档讲**为什么**。事实在 `SKILL.md`，用法在 `README.md`，设计在 `DESIGN.md`。

---

## 1. 这个 skill 解决什么问题

**真正的问题不是"Markdown 转 HTML"，是"让公众号文章每次都有高级感"。**

写作者的现实链路：

1. 写完一篇 Markdown 文章。
2. 想发微信公众号，需要排版成编辑器可粘贴的 HTML。
3. 用在线 Markdown 转 HTML 工具，结果每次都长得不一样。
4. 用 Canva 或 135 编辑器套模板，被几百个相似排版挤掉品牌感。
5. 让 AI 生成排版，结果 AI 每次都"临场发明"一套新样式，和上次完全不同。

我们要在第 3-5 步之间插一个东西，让排版**确定性地**保持品牌感。

### 核心矛盾

- **质量**：要像手工精修的杂志正文版式，不能像默认 Markdown 渲染。
- **一致性**：同一篇文章用同一个主题渲染 10 次，结果必须一致。
- **可维护**：新增主题不能靠 AI 即兴发挥，必须有确定性的 token、组件规则和验证门槛。

三者之中**一致性是分水岭**。Canva 模板解决了质量。AI 生成解决了速度。但都没解决"同一套主题每次输出都一样"这件事。

## 2. 产品定位（一句话）

> 把一篇 Markdown 文章交给 AI agent，用确定性的语义组件和主题哲学，输出微信公众号编辑器可粘贴的高级感 HTML，**每次结果都一样**。

## 3. 目标用户与典型场景

主要画像是**"有内容但没有设计师的技术作者 / 产品经理 / 内容运营"**。共同点：自己写文章，没有"组里专门做排版的人"。

| 场景 | 用户类型 | 输入 | 期望输出 |
| --- | --- | --- | --- |
| 技术长文发布 | 工程师 / 技术博主 | Markdown 技术文章 | 宋式美学排版的 HTML |
| 产品方案分享 | 产品经理 | Markdown 方案文档 | 科技主题排版的 HTML |
| 方法论文章 | 咨询顾问 / 作者 | Markdown 方法论长文 | 稳境白纸主题的 HTML |
| 知识科普 | 研究员 / 科普作者 | Markdown 科普文章 | WISME 规范主题的 HTML |

**反向画像**（这个 skill 不服务他们）：

- 想做动态交互页面：我们只输出静态内联 HTML。
- 想做 PPT / 演示文稿：这是正文排版，不是幻灯片。
- 想自定义颜色 / CSS / 主题文件：我们不允许，保护美学比给自由更重要。
- 想生成封面图 / 标题图 / 账号信息：这些是微信平台功能，不是正文排版。

## 4. 关键设计决策（为什么这样做）

### 决策 1：确定性优先于 LLM 即兴发挥

v2 之前，每次渲染都依赖 LLM 临场决定字号、间距、颜色和组件样式。结果是：同一篇文章每次输出都不一样，无法保持品牌一致性。

v2 的解法：把所有视觉行为编码到主题 JSON 和 renderer 脚本中。LLM 只做编辑判断（结构、节奏、重点句、语义组件选择），不做视觉判断（字号、间距、颜色、布局）。

**代价**：用户不能传自定义 CSS，灵活性降低。
**收益**：同一篇文章每次输出完全一致，品牌感可复现。

### 决策 2：不允许自定义主题色

用户传一个 `#A3FF21` 进来，AI 配的衍生色大概率难看。当前 4 套主题是经过视觉验证的上限。

宁可让用户从 4 套里选一个最接近的，也不让用户精确指定。和"不接动态页面"是同一类决策——**保护美学 > 给自由**。

### 决策 3：语义组件不是装饰模块

`:::lead` 不是"给段落加个好看的背景"，而是"这段内容在公众号正文中的最佳表达方式是导语"。

语义组件回答的问题是"这段内容最适合如何被读者接收"，而不是"这里要加什么装饰"。这决定了组件的设计逻辑：先有语义，再有视觉。

### 决策 4：配图由宿主 agent 负责

skill 不配置生图模型、不保存 API key、不绑定 provider。原因：

1. 生图能力是平台能力，不是排版能力。把两者绑定会增加 skill 的复杂度和维护成本。
2. 不同用户的生图环境不同（有的用 GPT-Image，有的用 Midjourney，有的没有生图能力）。skill 应该与生图解耦。
3. skill 只负责定义配图标准（比例、必要性、主题契合度）和验证 image-plan.json，不负责执行生图。

**代价**：用户需要宿主 agent 有生图能力才能自动配图。
**收益**：skill 保持轻量，不依赖外部 API，可在任何环境运行。

### 决策 5：输出到 ~/.wechat-typeset/ 而非 skill 目录

用户产物写入 skill 目录会导致：
- skill 更新时可能覆盖用户产物。
- 用户产物和 skill 代码混在一起，难以清理。
- 多用户共享 skill 目录时产物互相干扰。

写入 `~/.wechat-typeset/<slug>/vNNN/` 解决了这三个问题。

### 决策 6：v2 重构为语义组件系统

v1 是"Markdown → HTML"的直接映射，每个主题单独实现渲染逻辑。问题是：新增主题需要重写整个 renderer，成本高且容易不一致。

v2 的解法：建立共享的语义组件矩阵（COMPONENTS.md），每个主题只实现组件的视觉差异（字号、间距、颜色、装饰），不改变组件的语义可用性。新增主题变成"翻译"而非"重写"。

## 5. 产品边界（我们不做什么）

| 不做 | 原因 |
| --- | --- |
| 不允许自定义颜色 / CSS / 主题 | 保护美学一致性 |
| 不生成封面图 / 标题图 | 这是微信平台功能 |
| 不控制文章标题 / 账号信息 / 发布流程 | 只管正文排版 |
| 不输出 PPT / PDF / 动态页面 | 只输出微信兼容的静态内联 HTML |
| 不在 skill 内调用生图模型 | 生图是宿主 agent 的能力 |
| 不允许运行时新增主题 | 主题必须经过设计确认和验证 |
| 不追求桌面表格原貌 | 微信移动端安全优先 |

## 6. 反模式（我们主动拒绝的事）

1. **LLM 临场写样式**：让 AI 直接生成 HTML / CSS / style / class → 拒绝。所有样式由主题 renderer 决定。
2. **自定义主题色**：用户传任意 hex 值 → 拒绝。只能从 4 套预设选。
3. **装饰性语义组件**：把 `:::lead` 当成"好看的背景块" → 拒绝。组件是语义选择，不是装饰。
4. **伪造配图证据**：生成图片伪造截图、数据、图表、品牌、引用 → 拒绝。evidence_figure 必须有 source_note。
5. **过度强调**：highlight 占比超过 8%、连续 emphasis 超过 2 段 → 拒绝。克制是高级感的核心。
6. **移动端溢出**：表格依赖横向滚动、代码块超出屏幕 → 拒绝。faux table 和截断代码是必须的。
7. **空泛模块堆叠**：用"流程重设计""标准成为杠杆"这类可套任何主题的空话填充 → 拒绝。内容必须具体。
8. **标题跳级**：H1 直接到 H3，跳过 H2 → 拒绝。heading hierarchy 禁止跳级。

## 7. 度量方向

| 指标 | 目标值 | 检查方式 |
| --- | --- | --- |
| 渲染一致性 | 同一输入 × 同一主题 = 相同输出 | 内容 hash 对比 |
| Highlight 占比 | <= 8% | audit_visual.py |
| 连续长段落 | <= 3 段 | editorial_verify.py |
| 连续 emphasis | <= 2 段 | editorial_verify.py |
| 移动端横向滚动 | 0 | html_verify.py |
| 主题扩展时间 | 新主题 < 2 天 | 人工记录 |
| 验证通过率 | 100% | check_all.py |

## 8. 未来路线图

按 ROI 排序。

### 短期（下两轮内）

1. **量化密度标准**。把"连续长段落 <= 3"等规则转化为可被脚本自动检查的量化指标。
2. **Seed 模板机制**。为每个主题提供预渲染的 seed HTML，agent 只需注入内容而非从零构建。
3. **用户审查流程**。渲染后先给用户看，问要不要跑 validator，而不是自动跑。

### 中期

4. **更多主题**。按 METHODOLOGY.md 的快速管线扩展，优先覆盖更多文章类型。
5. **配图工作流简化**。如果宿主 agent 有生图能力，自动从 image-plan.json 生成图片。
6. **跨平台输出**。考虑输出适配其他平台（知乎、掘金、Medium）的 HTML 变体。

### 长期

7. **Web UI 微调**。渲染后让用户在浏览器里微调文案/换图/调间距，再触发重渲染。不做"从零拼版"。
8. **品类专属配图 prompt 库**。按文章类型预设配图风格和提示词。

### 真的不做

- 风格自定义 / 用户主题色：保护美学 > 给自由。
- 动态交互页面：只做静态内联 HTML。
- PPT / PDF 导出：这是正文排版，不是演示工具。
