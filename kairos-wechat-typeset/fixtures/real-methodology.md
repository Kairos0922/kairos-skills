# 产品设计中的约束思维

在产品设计中，约束不是限制，而是创造力的来源。

==最好的设计系统不是给用户无限自由，而是替用户排除绝大多数会变丑的选项。==

## 01 为什么约束比自由更重要

大多数人的直觉是：给用户更多选择 = 更好的产品。但实际经验告诉我们相反的结论。

当用户面对 100 种颜色选择时，他们反而不知道选什么。当用户面对 10 种预设时，他们能快速找到最合适的。

> [!NOTE] 约束的本质
> 约束不是剥夺自由，而是降低决策成本，提高输出质量。

### 自由的代价

- **决策疲劳**：太多选择导致用户放弃
- **质量失控**：没有约束的输出往往平庸
- **维护成本**：每种组合都需要测试和验证

### 约束的收益

- **一致性**：所有输出保持统一的品质
- **效率**：用户快速做出决策
- **可维护**：系统状态空间有限，容易测试

## 02 设计系统中的约束层次

一个好的设计系统有多层约束：

### 硬约束（不可违反）

这些是系统级别的规则，任何情况下都不能违反：

- 颜色只能从预设中选择
- 字号只能使用 token 定义的值
- 布局只能使用预定义的骨架

### 软约束（有理由时可调整）

这些是推荐规则，在特殊情况下可以偏离：

- 行高推荐 1.7，但某些场景可以用 1.5
- 间距推荐 24px，但某些场景可以用 16px

### 引导（建议但不强制）

这些是最佳实践，不是硬性要求：

- 标题应该比正文大
- 引用块应该有呼吸感
- 表格后应该有解读文字

## 03 约束的实现方式

### 代码约束

用代码实现的约束是最可靠的：

```python
# 这是硬约束——用户无法绕过
ALLOWED_COLORS = ["#2b2b2b", "#7a7a72", "#f7f5f0", "#8b7355"]
```

### 配置约束

用 JSON 配置实现的约束是灵活的：

```json
{
  "title_size": "30px",
  "body_size": "16px",
  "body_line": "1.7"
}
```

### 文档约束

用文档实现的约束是最弱的，但覆盖面最广：

```markdown
## 禁止事项
- 不要使用渐变背景
- 不要超过 3 种功能色
```

## 04 约束与品味的关系

约束是品味的编码。

当你说"不要用渐变"，你其实是在说"我认为平涂更高级"。当你说"不要超过 3 种颜色"，你其实是在说"我认为克制更优雅"。

> [!NOTE] 品味不是玄学
> 好看不是玄学，而是可拆解、可编码、可检查的行业常识。

约束把这些常识从个人脑子里提取出来，变成系统可以执行的规则。这样，即使没有品味的 AI，也能产出有品味的结果。

## 05 实践案例

### 案例 1：颜色约束

**问题**：AI 生成的图表经常使用 5-6 种颜色，看起来像调色板而不是设计。

**约束方案**：
- 只允许 4 种功能色：主色、辅色、背景、强调
- 强调色覆盖率 <= 8%
- 不允许渐变

**结果**：所有输出自动保持颜色克制。

### 案例 2：排版约束

**问题**：AI 生成的文章标题和正文字号差异太小，层次感不足。

**约束方案**：
- 标题字号固定 30px
- 正文字号固定 16px
- 比例 1.875:1，层级清晰

**结果**：所有文章自动有清晰的视觉层级。

### 案例 3：组件约束

**问题**：AI 不知道什么时候用引用块、什么时候用洞察块。

**约束方案**：
- 定义 6 种语义组件，每种有明确的使用场景
- 引用块只用于外部引用
- 洞察块只用于核心判断

**结果**：AI 只需要判断"这段内容是什么"，不需要判断"用什么样式"。

## 06 结语

约束是产品设计中最被低估的工具。它不是限制创造力，而是释放创造力——让 AI 把精力花在内容上，而不是样式上。

好的设计系统 = 好的约束。
