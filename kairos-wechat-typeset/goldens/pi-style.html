用 Pi 构建你的第一个 AI Agent

好的工具不是让你做更多事，而是让你做更少的事，但每一件事都做到位。

原创　Pi Dev　｜　2026.06.18

Pi 是一个极简的 agent harness。它的哲学很简单：不要替用户决定工作流，让用户定义自己的工作流。 旧的 all-in-one 模式正在被取代，可组合的模块化工具才是开发者的未来。

Pi 不是另一个 AI 编程助手。它是一个让你构建自己的 AI 编程助手的框架。

01为什么选择极简

大多数开发工具的逻辑是：功能越多越好。结果是界面臃肿、选项泛滥、学习曲线陡峭。

Pi 的逻辑相反：

1.核心只做四件事：读文件、写文件、编辑文件、跑命令

2.其他一切通过扩展、技能、包来添加

3.你需要什么，就装什么

先用默认配置跑起来，遇到限制再扩展。过早优化配置是开发者最常见的陷阱。

02五分钟上手

安装只需要一行命令：

BASH

npm install -g --ignore-scripts @earendil-works/pi-coding-agent

然后进入你的项目目录：

BASH

cd /path/to/your/project pi

步骤 × 动作 × 时间

步骤安装动作npm install -g时间30 秒

步骤启动动作cd project && pi时间5 秒

步骤第一次对话动作输入你的需求时间随时

--ignore-scripts 是故意的。Pi 不需要安装脚本，这个标志跳过依赖的生命周期脚本。

03上下文工程

Pi 的核心竞争力不是模型，而是上下文控制。

其他 agent 把所有东西塞进 system prompt。Pi 让你精确控制什么进入上下文窗口，什么时候进入，以什么顺序进入。

Pi 加载上下文的顺序：

1.~/.pi/agent/AGENTS.md — 全局指令

2.父目录的 AGENTS.md — 项目级指令

3.当前目录的 AGENTS.md — 本地指令

4.SYSTEM.md — 替换或追加系统提示

○按需加载技能，不一次性塞满上下文

○自动压缩旧消息，避免超出窗口限制

○支持自定义压缩策略：按主题、按代码感知、用不同模型做摘要

04四种模式

Pi 不是一个模式打天下。它有四种模式适应不同场景：

1.交互式：完整的 TUI 体验，适合日常开发

2.Print/JSON：pi -p "query" 一行命令，适合脚本集成

3.RPC：JSON 协议 over stdin/stdout，适合非 Node 集成

4.SDK：嵌入你的应用，适合构建产品

不要让工具决定你的工作流。让工作流决定你需要什么工具。

05扩展生态

Pi 的扩展系统是 TypeScript 模块，可以访问工具、命令、快捷键、事件和完整的 TUI。

扩展、技能、提示模板和主题都可以打包为 Pi 包，通过 npm 或 git 分享。

常用扩展场景：

•[ ] 子 agent 调度

•[x] 权限网关

•[x] 路径保护

•[ ] 沙箱执行

•[x] MCP 集成

极简不是缺少功能，而是每个功能都有存在的理由。Pi 把选择权还给了开发者。
